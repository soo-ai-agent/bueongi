# 기획 검토 — 스프린트 로드맵

> 작성: 기획 검토(Planner Review) · 대상: 안심귀가(부엉이) MVP
> 범위: 프론트 `bueongi` + 백엔드/파이프라인 `bueongi-service`

안심귀가 MVP를 **기획자 시선**에서 검토한 결과를, 실행 가능한 수정사항으로 묶어
스프린트 단위로 정리한 문서 모음이다. 각 스프린트 파일은 "무엇을 바꾸는지 + 완료 기준(AC)"
형태의 백로그로 구성된다.

## 전제 — 인정하고 가는 강점

- **시크릿 위생(secret hygiene)**: 클라이언트 노출 가능한 키만 allowlist로 좁히고
  (`src/app/utils/clientEnv.ts`), server-only 패턴은 빌드 단계에서 차단
  (`clientEnvBuildGuard.ts`, NF5). API 키를 저장소에 올리지 않은 것은 **의도된 보안 설계**다.
- **정직성 원칙**: 가짜 안내 금지 / 무손상 폴백 / 정직한 공유 상태.
- **명확한 MVP 범위 + v2 제외 정의**, **요구사항↔테스트 추적 매트릭스**.

> 키가 없을 때 목업/폴백으로 동작하는 것은 결함이 아니라 설계된 정상 동작이다.
> 따라서 기획의 초점은 "왜 샘플이냐"가 아니라 아래 세 가지로 이동한다.
> ① 운영 중 폴백이 켜져도 사용자가 속지 않게 하기(라이브/폴백 투명화)
> ② 번들 노출 키의 남용·쿼터·회전 거버넌스
> ③ 환경별 키 주입 운영 파이프라인

## 우선순위 원칙

- **P0**: 키 유무와 무관하게 안전 신뢰에 직결 — 출시 전 필수.

- **P2**: 믿고 쓰고 지속 가능한가 — 프라이버시·법규·수익화·측정.
